Vault 底层逻辑
总论
网站由两个部分组成：

数据
Skills
我们希望，对于“数据”，尽可能减少主动上传，而是在用户使用时收集数据。对于“skills”，尽可能的自定义化。

数据
学生的数据包括：

问卷答案
聊天记录
自己上传的文件
Lab 代码库（注意不是文件，而是一个代码库）
老师数据包括：

讲义
教材
题库
。。。
通过这些数据，我们最终要为每个学生构建一个肖像。我们的诸多功能，都是基于这个肖像去做的。

比如说当老师去询问“哪个学生最积极”，大模型本质是在查找最符合“积极”这个标准的学生肖像。

Skills
Skills 就是用 markdown 表示的 agent workflow。

但是无论 skills 怎么样灵活，都无法摆脱数据的范畴。如果数据里面没有收集学生的代码仓库信息，那么老师再怎么精确的去利用“谁的代码风格最好”，也是没有效果的。


Vault Requirement
Vault（穹隆）是专门为 CS 教学开发的 Agent。

Basic
Vault 的本质是一个知识图谱（Knowledge Graph, KG）原生的 CS 教学 Agent，其呈现形式是一个 Web 网站。

在 Vault 中，节点有多种类型：

资料：教材、讲义、录播、试卷等
个人：成绩、作答、Lab 代码、提问与回答等
这些节点存在多种联系，比如“鸭鸭的期中试卷的答案”、“Lab1 对应教材的内容”等。 这些节点和联系共同组成了一个知识图谱。

Vault 会维护这个知识图谱，并根据这个知识图谱完成学习和研讨。

Functions
提问
用户可以像与常见 chatbox 聊天的模式，与 Vault 进行问答。

Vault 有以下特色：

会根据知识图谱中的资料进行回答
在回答的时候会进行资料的联想：问题 -> 知识点 -> 关联教材
在回答时会记录学生的信息，分析学生的水平
代码分析
Vault 可以分析学生的代码仓库，并更新学生肖像，获得的信息有：

学生潜在的代码 bug
学生的工程不规范的现象
学生使用 AI 的痕迹（AI 生成、AI 洗稿等）
问卷生成
Vault 可以根据特定的知识点和学生的肖像，生成针对性的题目。

学生知识追踪（Knowledge Tracing）
Vault 会跟踪学生的 Lab 代码、课堂小册答案、考试答案和提问等数据，分析学生哪个知识点存在薄弱环节。

分析的结果可以用于问卷生成，也可以单独供教师查看。

其他基础功能
账户管理功能
多功能协作：也就是在一个对话中，自动化使用多种功能
多模型选择与协作
肖像年度报告
Typical Scenarios
技术提问
鸭鸭是一名刚进入便宜坊的新生，老师上课教授的闷炉烤鸭的技术要点它没有学会。 于是它向 Vault 提问，“闷炉烤鸭最重要的是什么？”。

Vault 会：

检索一下《便宜坊食谱》，回答“闷炉烤鸭最重要的是不能有明火”
将《便宜坊食谱》的第 3 章推荐给鸭鸭
记录：“鸭鸭同学不清楚焖炉烤鸭技术”
将《闷炉烤鸭的 13 个优点》这类相关性很强的文章发送给鸭鸭
鸭鸭不仅知道了答案，而且知道了这个具体知识点的出处，还拓展了新的知识点。并且鸭鸭的个人肖像更加丰满了。

代码分析
鸭鸭不仅会做卷子，还会实操烤鸭子（就是写代码）。

Vault 会检验烤好的鸭子（就是代码库）：

观摩烤鸭子过程
发现虽然鸭鸭的闷炉守法确实没有明火，但是鸭鸭也没有使用木炭啊，鸭鸭使用的是酒精灯
更新鸭鸭的肖像。
问卷生成
鸭鸭要进行知识检测了，它希望通过问卷查漏补缺。

Vault 会：

调取鸭鸭的肖像，发现鸭鸭不清楚闷炉烤鸭技术，而且不清楚具体的燃料是什么
生成考题“焖炉烤鸭技术不能有明火的原因是什么？”
生成考题“闷炉烤鸭该用木炭还是酒精灯？”
校验鸭鸭的答案
根据鸭鸭的答案更新肖像。
肖像分析
教师和鸭鸭都可以查看学生的肖像。

Vault 会：

耄耋在“荷叶饼和烧饼的区别”这个知识点上屡次犯错
鸭鸭凌晨 3 点了还在努力烤鸭子，是努力鸭鸭
鹅鹅喜欢全聚德胜过便宜坊，说明知识存在严重漏洞
在学期结束的时候，Valut 给鸭鸭发送了一个邮件：

“聪明的鸭鸭，在烤鸭这门课里面，你一共烤了 514 只鸭子，被你解决的薄弱知识点有 114 处，……（参考网易云年度报告）”


Vault UI 界面
学生端
学生看板
学生可以在这个界面看到一些固定的信息，比如说：

学号
问问题的数目
其他比较固定的信息（我觉得之前的就挺好）
问卷分数和排名等
学生也可以自定义一些信息显示： 比如 “我最薄弱的知识点排行，列出 5 个”。 然后 LLM 就会根据这个问题，展示出对应的信息。

学生可以把这样的“卡片”粘在看板上。“卡片”会实时更新。

智能问答
就是一个普通的 chatbox 功能，用户可以向大模型提问。 大模型会根据知识库给出答案，在给出答案的同时，还会给出参考资料出处。 而且，大模型还会根据问题，合适的调用 skills，辅助对于问题的讲解。

用户也可以向 llm 上传代码或者 pdf 教材文件等。

用户也可以分享对话。

问卷
学生可以在这个地方做问卷（类似于那种普通的电子问卷的交互逻辑）。

教师端
教师看板
教师也可以在这个界面看到一些固定的信息，比如说：

成绩
教师同样可以自定义一些信息显示： 比如：

这个班最不了解进程概念的同学是谁？
这个班最活跃的五名同学是谁？
这个班代码风格最好的同学是谁？
这个班谁的代码和成绩谁最不匹配？
这些信息也会实时更新（相当于以一定时间间隔一直在问学生肖像）

智能问答
就是教师直接提问班上的情况（目前的 demo 就挺好的）。

问卷生成
教师会提供一套出题思路，比如说：

基础题目的占比是 50%
重点考查一下进程调度方面
针对每个人的薄弱点出 1 道题目
对于那些 AI 查重率高的同学，重点考查他的代码知识
教师也可以直接使用定义好的 skills 进行出卷（其实上面的本质上也是一个 skill）。

这个页面也可以展示生成好的问卷，方便老师检阅。

资料上传
教师可以向网站提交资料：

题库
教材
讲义
等等。

Skills 页面
教师会提供一套符合一定标准的 SKILL.md 文件或者更复杂的 SKILL 包（类似于 Anthropic 开发的 skill 格式）

具体的 SKILL Demo 可以参考这个链接里的文件。

这些 SKILL 会在需要的时候发挥作用，比如说学生的聊天中，或者问卷的生成过程中。

这个页面除了上传 Skill 以外，还要提供一些基础的管理功能。比如删除，展示等。
