# 基于知识库的问卷生成器

基于课程知识库内容生成问卷，结合已有文档和AI理解，生成紧密贴合课程内容的测试题。

## 适用场景

当用户需要：
- 基于上传的课程文档生成测试题
- 确保题目与教学内容高度相关
- 利用知识库中的实际知识点出题
- 生成针对性强的课程考核

## 知识库问卷生成规范

### 生成流程

1. **知识检索**: 已为你检索相关的课程文档内容
2. **内容理解**: 深入理解检索到的知识点
3. **题目生成**: 基于实际内容创建问题
4. **答案验证**: 确保答案来源于知识库内容

### 输出格式要求

必须严格遵循以下JSON格式：

```json
{
  "survey_title": "基于课程内容的问卷标题",
  "description": "问卷描述，说明基于哪些知识点",
  "knowledge_sources": ["引用的知识库文档1", "文档2"],
  "questions": [
    {
      "question_type": "choice|judge|essay",
      "question_text": "题目内容",
      "options": ["选项数组"],
      "correct_answer": "正确答案",
      "score": 分数,
      "explanation": "详细解析",
      "knowledge_source": "该题来源的知识点/文档片段"
    }
  ]
}
```

### 题型规范（同AI生成）

#### 1. 选择题 (choice)
- **question_type**: "choice"
- **options**: 只写选项内容，**不要**在内容前加 "A."、"B."、"C."、"D."。例如：["32学时", "48学时", "64学时", "80学时"]。系统会按顺序显示为 A/B/C/D。
- **correct_answer**: "A" | "B" | "C" | "D"（表示第 1/2/3/4 个选项）
- **score**: 3-5分
- **knowledge_source**: 标注该题基于的知识库内容

#### 2. 判断题 (judge)
- **question_type**: "judge"
- **options**: ["正确", "错误"]（仅这两个词，不要加 "A."、"B." 等前缀）
- **correct_answer**: "正确" | "错误"
- **score**: 2-3分
- **knowledge_source**: 标注依据的知识点

#### 3. 问答题 (essay)
- **question_type**: "essay"
- **options**: []
- **correct_answer**: 基于知识库的详细参考答案
- **score**: 8-15分
- **knowledge_source**: 标注涉及的知识范围

### 知识库问卷特殊要求

1. **只出“知识点”题，完完全全基于知识点（最高优先级）**
   - **“知识点”仅指**：概念、定义、原理、技术名称、操作技能、章节中的**技术/理论要点**。例如：进程、死锁、Vibe Coding、分布式数据管理、某实验的目标与步骤（技术部分）等。
   - **必须**每道题直接考查一个或多个上述**具体知识点**，题干直接问该知识点本身，例如：「进程是什么？」「下列哪项属于进程的三种基本状态？」「Vibe Coding 是什么？」「分布式协同与跨端通信实验的目标包括哪些？」。
   - **禁止**基于以下内容出题（即使出现在检索结果中也要忽略）：
     - 课程基本信息/元信息：总学时、学分、先修课程、课程代码、任课教师、开课学期；
     - 思政融入点、课程思政要求、育人目标；
     - 考核说明、考核方式、考核原则、成绩评定、提审上线规定、伪造处理；
     - 课程要求、学生行为规范、能否过度依赖AI、独立完成要求等**非技术性规定**；
     - 课程简介中与具体技术/概念无关的概括性描述。
   - **题干表述**：禁止用「根据课程内容」「课程中，学生可以…吗」「本课程」「本课程中」「目标要求」「学生」「要求学生」「考核学生」「训练哪项技能」「考核…能力」「…实验/实训的目标」「目标之一」「旨在…」「学生需要掌握」「核心技能要求」等字样；直接写对知识点的提问，只问**技术或知识点本身**。**禁止**以「…实验/实训的目标要求」「…的目标是什么」等形式出题；若考查某实验或实训，须问其**技术内容**（如「…实验主要涉及哪些技术？」），不要问「训练哪项技能」「考核学生哪些能力」「…的目标是什么」。
   - 错误示例（禁止）：「本课程总学时是多少？」「分布式协同与跨端通信实验**要求学生**开发分布式任务调度模块。」「组件开发与跨端适配实践实验主要**训练哪项技能**？」「开源平台实践环节主要**考核学生**哪些方面的能力？」「应用提审规范学习与上线准备实训的**目标之一**是…」「简述…实验的目标要求。」「学生需要掌握哪些核心技能？」。
   - 正确示例：「进程的三种基本状态是什么？」「什么是Vibe Coding？」「Flutter主要用于开发鸿蒙应用的哪一层？」「分布式协同与跨端通信实验主要涉及哪些技术？」「组件开发与跨端适配实践主要涉及哪些技术？」「简述实现分布式协同与跨端通信需掌握的技术要点。」「开源平台实践主要涉及哪些技术规范与能力？」。

2. **题目表述（必须遵守）**
   - **question_text 只写题目本身**，禁止「基于xx生成」「根据xx生成」等前缀；**禁止**题干中出现「本课程」「本课程中」「学生」「要求学生」「考核学生」「训练哪项技能」「考核…能力」「目标要求」「…实验/实训的目标」「目标之一」「旨在…」「学生需要掌握」「核心技能要求」等字样；只问技术/知识点本身，禁止问「训练哪项技能」「考核学生哪些能力」「…的目标是什么」。
   - 正确示例：`"question_text": "下列哪项属于进程的三种基本状态？"`、`"question_text": "Flutter 主要用于开发鸿蒙应用的哪一层？"`、`"question_text": "分布式协同与跨端通信实验主要涉及哪些技术？"`、`"question_text": "组件开发与跨端适配实践主要涉及哪些技术？"`
   - 错误示例：`"question_text": "分布式协同与跨端通信实验要求学生开发分布式任务调度模块。"`、`"question_text": "组件开发与跨端适配实践实验主要训练哪项技能？"`、`"question_text": "开源平台实践环节主要考核学生哪些方面的能力？"`、`"question_text": "应用提审规范学习与上线准备实训的目标之一是…"`

3. **数量与题型默认**
   - 若用户描述中未说明题目数量和题型，默认生成 **20 道题**，题型为选择题、判断题、问答题三种合理分布。

4. **内容准确性**
   - 所有题目必须**严格基于**知识库中的知识点和知识图谱，不能编造
   - 答案必须与知识库内容完全一致
   - 不能基于常识或网络知识，只能基于提供的检索内容
   - 如果知识库内容不足，宁可少出题
   - 每个知识点只出1-2道题，避免重复

5. **知识覆盖（基于大纲时必须满足）**
   - 基于课程大纲生成时：检索到的**每一章、每个实验、每个实训项目等至少对应一题**，确保整套题**覆盖全部**检索到的知识点、无遗漏；这样教师才能依据答题情况判断学生对整门课的掌握程度。不得把多道题集中在少数章节。
   - 优先使用检索到的核心内容，题目应覆盖多个不同的知识点，避免题目重复或过于集中在一个知识点
   - 确保知识点分布均匀，尽量从不同文档片段、不同章节出题

6. **难度把控**
   - 根据知识库内容的深度确定难度
   - 基础概念、定义用选择/判断题
   - 复杂内容、综合理解用问答题
   - 难度要符合课程教学阶段
   - 有简单→中等→困难的梯度

7. **答案解析（explanation）**
   - 解析必须与**正确答案一致、确保正确**，不得与答案矛盾。
   - 解析中**禁止**出现「学生」「要求学生」「让学生」「培养学生」「提升学生」「考核学生」「该实验/实训的目标」「旨在」「核心目标是」「旨在培养」「旨在提升」等字样；解析应**直接说明技术要点或答案依据**，不要描述教学目的、学生行为或考核目的。
   - 解析中**不要**出现「根据课程大纲」「根据教学大纲」「根据大纲」「根据课程内容」等表述；可结合知识点与通用表述来写，简明扼要说明为什么该选项正确/错误。
   - 正确示例（解析）：「鸿蒙分布式能力包括分布式数据管理、多设备协同与分布式任务调度，是实现跨设备应用的基础。」「弹性布局、网格布局、相对布局等是 ArkUI 常用布局方式，用于构建自适应界面。」
   - 错误示例（解析）：「该实验的**核心目标是让学生**熟练运用…」「**旨在提升学生**运用现代工具…」「**要求学生掌握**不同平台的布局技术」「**旨在培养学生**遵循开源协作规范…」

8. **溯源标注（必须）**
   - 每道题必须标注 `knowledge_source`
   - 说明该题基于的具体知识点或文档段落
   - 格式：文档名 - 章节/知识点
   - 便于验证和追溯题目来源
   - 要具体到章节或段落

9. **格式统一性**
   - 选择题 options 只写内容，不要加 "A."、"B."、"C."、"D."，例如：["内容1", "内容2", "内容3", "内容4"]（必须4个选项）
   - 判断题 options：["正确", "错误"]
   - 分数总和必须精确等于100分
   - JSON格式必须完全正确，可解析
   - 不要有题目编号前缀

### 生成策略

#### 已提供知识库内容时

1. **先过滤再出题**：检索结果中可能包含课程基本信息、思政、考核说明、学生规范等。**一律不得用于出题**。只使用体现**具体知识点**的片段：概念、定义、原理、技术、实验目标与步骤（技术部分）、能力要求中的技术要点。
2. **识别可出题的知识点**：从剩余片段中识别**技术/理论知识点**（如：进程、Vibe Coding、分布式协同、某实验的技术目标），忽略“考核”“思政”“课程要求”“学生可以/不可以”等表述。
3. **只出知识点题**：每道题必须直接考查上述某个知识点，题干直接问该知识点（如「X是什么？」「下列哪项是X的特点？」），不要出「本课程…」「学生…」「要求学生」「考核学生」「训练哪项技能」「目标要求…」「…实验的目标」「…实训的目标之一」「旨在…」「学生需要掌握…」「课程中…吗」「考核…」「根据课程内容…」类题目；实验/实训类须问技术内容，不问「目标」或「训练/考核学生」。
4. 每个知识点最多出2道题，答案必须从知识库片段中可找到依据；题干和解析均**禁止「学生」及教学/考核视角**（如 要求学生、考核学生、旨在培养、该实验的目标）；解析只写技术要点或答案依据；检索到的各章节/实验/实训须全覆盖、至少各有一题。
```
知识库检索结果示例（仅使用知识点类片段出题）：
[文档片段: 进程的三种基本状态：就绪、运行、阻塞...]  → 出题「下列哪项属于进程的三种基本状态？」
[文档片段: Vibe Coding 是实时协作开发模式...]        → 出题「什么是 Vibe Coding？」或「Vibe Coding 指什么？」
[文档片段: 考核说明：提审材料不得伪造...]            → 忽略，不出题
[文档片段: 思政融入点：学生需独立完成核心逻辑...]    → 忽略，不出题
```

#### 题目设计原则

- **选择题**: 测试概念理解、细节记忆
  - 题干来自知识库核心内容
  - 正确答案严格基于文档
  - 干扰项设计要合理但明确错误
  - 选项内容要来自或基于文档

- **判断题**: 测试关键结论、定义准确性
  - 陈述应直接引用或改写知识库内容
  - 避免模棱两可的表述
  - 可以是对知识库内容的否定（作为错误选项）

- **问答题**: 测试深度理解、综合应用
  - 要求学生组织多个知识点
  - 参考答案整合知识库相关内容
  - 给出明确的评分要点
  - 答案必须完全基于文档内容

### 质量检查清单

生成问卷前必须确认：
- [ ] 所有题目都基于检索到的知识库内容
- [ ] 每道题都标注了 `knowledge_source`
- [ ] 答案可以在知识库中找到依据
- [ ] 没有编造或臆测知识库外的内容
- [ ] JSON格式完全正确
- [ ] 分数分配合理，总分精确等于100
- [ ] 解析详细且基于实际内容
- [ ] 题型符合用户要求（只生成指定题型）
- [ ] 选择题选项只写内容，不要 "A."、"B." 等前缀
- [ ] 判断题选项：["正确", "错误"]
- [ ] 没有题目重复
- [ ] 知识点分布均匀

### 重要提醒

- **只出知识点题**: “知识点”仅指概念、定义、原理、技术、技能；禁止根据思政、考核说明、课程要求、学生规范、课程元信息出题；题干直接问知识点（如「进程是什么？」），不要「根据课程内容」「课程中，学生可以…吗」
- **必须完完全全基于知识点**: 每道题对应知识库中的具体知识点，不得编造；检索到某知识点则围绕该知识点出题
- **题目直接写题干**: question_text 禁止「基于xx生成」「根据课程内容」等前缀，只写对知识点的提问
- **数量默认**: 描述未说明题型和数量时，默认 20 道题、三种题型
- **标注来源**: knowledge_source 须为具体知识点/章节，不能为考核说明、思政、课程基本信息
- **内容一致**: 答案必须与知识库内容完全一致
- **只输出JSON**: 不要有markdown标记或其他文字；分数总和恰好100分
